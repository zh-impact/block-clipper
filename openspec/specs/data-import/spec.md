## Purpose

Define the baseline behavior for importing Block Clipper JSON exports, including file validation, record validation, deduplication, result reporting, and storage integrity guarantees.

## Requirements

### Requirement: System SHALL import Block Clipper JSON export files
The system SHALL allow users to import JSON data generated by Block Clipper export from file input and clipboard input.

#### Scenario: Import valid exported JSON file
- **WHEN** user selects a valid Block Clipper export JSON file and confirms import
- **THEN** system SHALL parse the file and proceed to validation/import flow

#### Scenario: File is not JSON
- **WHEN** user selects a non-JSON or unreadable file
- **THEN** system SHALL reject import and show an actionable error message

#### Scenario: Import valid exported JSON from clipboard
- **WHEN** user triggers import-from-clipboard and clipboard contains valid Block Clipper export JSON
- **THEN** system SHALL parse clipboard text and proceed to validation/import flow

#### Scenario: Clipboard content is invalid
- **WHEN** user triggers import-from-clipboard and clipboard text is non-JSON, incompatible JSON, or unreadable
- **THEN** system SHALL reject import and show an actionable error message

### Requirement: System SHALL validate imported record structure
The system SHALL validate each parsed record against required import fields before persistence.

#### Scenario: Record missing required fields
- **WHEN** a parsed record lacks required fields (`type`, `content`, `source.url`, `source.title`)
- **THEN** system SHALL mark the record as failed and SHALL NOT persist it

#### Scenario: Record contains unsupported block type
- **WHEN** a parsed record includes an unsupported block type value
- **THEN** system SHALL mark the record as failed and continue processing remaining records

### Requirement: System SHALL deduplicate during import
The system SHALL skip duplicates using stage-1 duplicate criteria and preserve existing data.

#### Scenario: Duplicate record detected
- **WHEN** an import record matches an existing record by duplicate criteria
- **THEN** system SHALL skip persistence for that record and count it as skipped

#### Scenario: Record is unique
- **WHEN** an import record does not match existing data
- **THEN** system SHALL persist the record to storage

### Requirement: System SHALL report import results
The system SHALL provide a user-visible summary after import across both file and clipboard channels.

#### Scenario: Import completes with mixed outcomes
- **WHEN** import run has imported, skipped, and failed records
- **THEN** system SHALL show counts for each outcome and include failure reason summary

#### Scenario: Import fully fails
- **WHEN** no records are imported due to file-level or clipboard-level validation/parsing failure
- **THEN** system SHALL show a clear failure message and SHALL NOT alter stored data

#### Scenario: Clipboard permission or API failure
- **WHEN** clipboard read operation is denied or unavailable
- **THEN** system SHALL show actionable feedback and SHALL NOT execute persistence operations

### Requirement: System SHALL preserve storage integrity on import errors
The system SHALL continue processing valid records even when some records fail validation.

#### Scenario: Partial record errors
- **WHEN** some records fail and others are valid
- **THEN** valid records SHALL still be imported and failed records SHALL be isolated in report output

#### Scenario: Unexpected runtime error during import
- **WHEN** an unexpected runtime/storage error occurs during import
- **THEN** system SHALL stop or safely continue per implementation policy and SHALL show actionable error feedback without corrupting existing data
